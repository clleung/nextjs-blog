Technical Writing Intern - Customer Experience: QuickSight Documentation Team

Role: Technical Writing Intern.

Tools: XML, Amazon QuickSight Q, Excel, Mockaroo.

Deliverables: Walkthrough Documentation, Project Artifacts, Data Model, Datasets, Amazon QuickSight Q Example, Presentation.

Skills: Information Architecture, Documentation Writing, Database Design.

Timeline: June - August 2021.

Location: Seattle, Washington.

Overview

I interned at Amazon Web Services (AWS) as a Technical Writing Intern and developed walkthrough documentation for QuickSight Q, a natural language query and deep learning feature that would be released later in 2021.

The walkthrough focused on streamlining the process of using the new feature for new and pre-existing customers of QuickSight, AWS’ Power BI tool. To support the walkthrough, I designed a database and simulated its integration into Amazon QuickSight Q to understand potential pain points when using the feature.

Post-internship, my documentation and project artifacts were used by the production team to refine the feature before release and informed publicly released documentation.

Goals

  1. Understand the intended roles of end-users and their touchpoints with the feature.
    • Accomplished by: Meeting the needs for authors and readers, along with their organizations.

  2. Align the documentation with internal and external stakeholder needs.
    • Accomplished by: Working with the product, technology, business intelligence, and customer experience teams.

  3. Understand the product, which was still in development at the time.
    • Accomplished by: Testing QuickSight Q with a database model.

Understood the use cases for Amazon QuickSight Q and scoped the project.

User Roles

Readers (of the data) need to...
  1. Create questions.

User Scenario

I created potential user stories and scenarios to guide the scope of the documentation.

Acceptance Criteria

From the user roles, the user scenario, and conversations with the product manager and documentation team, I determined the following acceptance criteria for the project. I factored in my interest in explaining technical concepts to non-technical audiences and lack of expertise with the product to explore how my documentation could enhance their understanding of Amazon QuickSight Q's capabilities.

Propose

Goals  •  Project Proposal

Proposed a walkthrough to meet the user scenario and leveraged knowledge from prior research.

Goals

From the user scenario, I developed the following goals for the walkthrough:

  1. Minimize time to apply concepts from documentation to Amazon QuickSight Q.
  2. Reduce technical knowledge needed and be applicable to a wide range of customers.
  3. Showcase the unique selling points of the feature: the ability to work with data and produce visualizations instantly and dynamically.

Propose

With those goals in mind, I proposed focusing on teaching customers how to make their datasets natural language friendly for Q in the walkthrough.
From research and collaboration with product team, the process for the user could be abstracted into three main parts.

  1. Insert data into Amazon QuickSight Q within a Topic.
  2. Add details to the data.
  3. Test the Topic.

Ideate

Documentation Outline  •  Design Decision

Outlined ideal user behavior and processes to be explained within the documentation.

Documentation Outline

After outlining the documentation, I found it difficult to write in a way that would meet the goals of the walkthrough and be able to describe the tasks in detail without additional searching from the user.

Design Decision

While exploring the feature, I determined that a data model would be helpful in learning the feature and allow for the abstraction of the complexities behind QuickSight Q (NLQ, deep learning, voice design).

Design

Draft Data Model • Draft Conceptual Model • Incorporating the Data Model into Documentation

Explored an example (created with an example data model and dataset) to support the walkthrough documentation and abstract technicalities from the user.

Draft Data Model

I proposed centering the example around an example music streaming service, as there would be a large amount of data publicly available and the proliferation of streaming services across the world.

Draft Conceptual Model

Due to the potential widespread use of the documentation, it was also significant to explore opportunities that would not result in potential legal issues, such as using a popular song in the dataset.

The choice to focus on a music streaming database also came from prior experience developing a Spotify database in SQL (case study), which would allow me to focus more on learning Amazon QuickSight Q and integrating the example into the walkthrough documentation.

Iterate

Updated Conceptual Model • Updated Logical Model • Applying the Data Model to Amazon QuickSight Q

Updated Conceptual Model

After talking to subject matter experts (SME) about Power Business Intelligence Analytics to understand the use cases more deeply, I pivoted to focus more on streaming data. Specifically, how often a song would be played at different points in time.

In theory, this use case would be easily extensible to other industries, such as e-commerce (purchases of an item), content creation (views over time), and general business cases (revenue/cost over time).

Updated Logical Model

As a result, the use of the data model was reduce from a more holistic view of the business—revenue streams and cost streams—to focusing on streaming traffic, which involved fewer entities than the original (5 tables instead of 7).

Applying the Data Model to Amazon QuickSight Q

Although I tried to obtain approval to use publicly available datasets, there were potential legal limitations, so I created datasets with Mockaroo.

This allowed me to purposefully inject trends into the data that would be searchable with Amazon QuickSight Q.

After testing Amazon QuickSight Q, I refined the process in the documentation once more:

  1. Anticipate questions.
  2. Create a topic.
  3. Update the topic to be natural language friendly.
  4. Test the topic.

Deliver

Presentation

Reflection

At the end of my internship, I delivered a final presentation to the product manager of Amazon QuickSight Q, a draft walkthrough, the example in Amazon QuickSight Q, the data model and datasets, and the project artifacts.

Next Steps

Reflection • Outcomes

Reflection

Given more time, I would have liked to create a walkthrough that could be independent from the example. This way, users would be able to dive deeper into the steps if needed, but otherwise be able to achieve their goal of getting the necessary information to complete the task as quickly as possible.

If I could redo this project, I would have spent more time talking with users to deeply understand the use cases behind using Amazon QuickSight Q, and how documentation could be written (or visualized) to help them understand the product without having to explain the complexity of the technology underlying it.

Takeaways from this project: as a novice technical writer, I was shown how well-written instructions and micro-visualizations could impact the user experience. Coming from a technology and design perspective, it was easier to imagine the bigger picture, but when I was documenting the process, I found it immensely difficult to whittle down a single task into tiny, tangible steps customized to a specific use case. From this project, I learned to not only look at the big picture for users (scenarios, personas, journeys), but also their real-time touchpoints with a product or service (how they get something done, where they go next, where that fits within their entire interaction with the organization).

Outcomes

Although the documentation I wrote was never published publicly, I was the first within the AWS Documentation team to integrate a database example into their documentation, as well as the first to write documentation for future writers to use if they so choose.

For the Amazon QuickSight Q documentation team, the data model and dataset I created were to be used in their production team for testing. I was also able to provide valuable insight for how I understood the product from a technical perspective.
